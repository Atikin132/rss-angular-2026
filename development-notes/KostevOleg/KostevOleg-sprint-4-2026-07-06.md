# Sprint 4 Project Deep Dive Notes - 2026-07-06
Сегодня решил погрузиться в проект немного глубже и посмотреть уже не на свои фичи, а на архитектуру и решения, которые использовали другие участники команды.

Первое, что всегда привлекало внимание и в течение написания фич это — работа с API Commercetools. В `commercetools-api.service.ts` сервис хранит access token, время его истечения и отдельный `tokenPromise`. На первый взгляд это кажется мелочью, но решение действительно хорошее. Если одновременно стартует несколько запросов, они не начинают независимо получать новый токен. Первый запрос создает `Promise`, а остальные просто ждут его завершения.

Еще понравилось, что этот сервис стал единой точкой входа для работы с Commercetools. Методы `request<T>` и `post<T>` скрывают всю инфраструктурную часть: авторизацию, формирование URL, добавление project key и bearer token. Благодаря этому остальной код просто запрашивает нужные данные и не думает о подобных деталях.

Потом посмотрел каталог и использование `@ngrx/signals`. Лично мне кажется, что здесь вполне можно было обойтись обычным сервисом на сигналах без дополнительной библиотеки. Но сам стор сделан аккуратно. В одном месте хранится состояние товаров, категорий, брендов, фильтров, загрузки и ошибок. Особенно понравилась функция `loadCategoriesIfNeeded`: если категории уже загружены, повторный запрос не отправляется.

В `loadProducts` тоже есть интересная деталь. Товары и категории загружаются параллельно через `Promise.all`, поэтому приложение не тратит время на последовательное выполнение двух независимых запросов. После этого товары сразу мапятся с использованием `categoriesMap`, и внутри приложения уже используются понятные названия категорий вместо идентификаторов из API.

Отдельно отметил работу с фасетами брендов. Список брендов формируется не вручную, а приходит из Commercetools через `productsData.facets`. Благодаря этому фильтры автоматически соответствуют данным каталога и не требуют ручной поддержки при изменении ассортимента.

Еще одно место, которое понравилось, — папка `commercetools/mapper`. API возвращает довольно сложные вложенные структуры: `masterVariant`, `attributes`, локализованные поля, цены в центах, информацию о скидках. Маппер преобразует все это в простой объект `Product`, с которым уже удобно работать в приложении.

Хорошее решение — вынести получение атрибутов в небольшой helper `getAttribute`. Благодаря этому код остается компактным и не приходится постоянно искать нужные значения через `attributes.find(...)`. Понравилось и то, что цены сразу переводятся из `centAmount` в обычные значения.

Еще обратил внимание на утилиту `build-filters.ts`. Она превращает объект с фильтрами каталога в query string для Commercetools. Получается, что компоненты и стор работают с обычной моделью `CatalogFilters`, а особенности синтаксиса запросов полностью изолированы в одной функции.

Интересной показалась и фабрика `create-selection.store.ts`. Вместо нескольких практически одинаковых сторов автор сделал одну фабрику, которая принимает ключ `localStorage` и возвращает готовый store с методами `toggle`, `add`, `remove` и `clear`. Такой подход избавляет от дублирования кода.

Там же понравилась небольшая, но полезная деталь — `safeLoadArray`. Чтение данных из `localStorage` обернуто в `try/catch`, поэтому даже если сохраненные данные окажутся поврежденными, приложение просто начнет работу с пустым массивом.

В `ProductCard` тоже видно хороший подход к организации компонентов. Карточка товара собрана из небольших независимых частей: отдельные компоненты отвечают за цену, рейтинг и кнопки действий. Сам `ProductCard` практически не содержит бизнес-логики и выступает как композиция этих элементов.

Еще цепанула реализация галереи товара. Выбранное изображение хранится через `linkedSignal`, который берет первое изображение из входного массива. Такое решение автоматически синхронизирует состояние с входными данными, поэтому при изменении списка изображений выбранная картинка тоже корректно обновляется.

По маршрутизации тоже осталось хорошее впечатление. Основные страницы работают внутри `MainLayoutComponent`, а тяжелые разделы приложения загружаются через `loadChildren` и `loadComponent`. Приятно, что вся команда придерживалась архитектурных договоренностей, принятых в начале проекта, поэтому структура маршрутов получилась единообразной независимо от автора конкретной фичи.
