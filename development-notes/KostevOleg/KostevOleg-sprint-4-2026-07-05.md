# Sprint 5 Global Error Handling - 2026-07-05

### Запись в дневник

Сегодня работал над реализацией глобальной обработки HTTP-ошибок в приложении. Основной задачей было централизовать отображение ошибок для пользователя, не дублируя одинаковую логику в сервисах и компонентах.

Был добавлен `errorInterceptor`, который подключается через `provideHttpClient(withInterceptors(...))` и перехватывает все ошибки `HttpClient`. Интерсептор сначала пытается получить сообщение, пришедшее от API (`message` или `error_description`), а если его нет, использует сообщение по умолчанию в зависимости от HTTP-статуса через `getDefaultHttpErrorMessage`. После этого ошибка отображается пользователю через toast и  пробрасывается дальше, чтобы сервисы и компоненты могли продолжать собственную обработку.

Для отображения уведомлений был создан `ToasterService`, инкапсулирующий работу с `MatSnackBar`, а также реализован кастомный `ToastComponent`, который получает данные через `MAT_SNACK_BAR_DATA` и отображает текст ошибки, иконку и кнопку закрытия. Для сообщений об ошибках были добавлены отдельные глобальные стили, чтобы они визуально отличались от обычных snackbar-уведомлений.

Во время ревью выяснился важный архитектурный нюанс. Не каждая HTTP-ошибка должна автоматически показываться пользователю. Например, отсутствие активной корзины (`404`) является ожидаемой частью логики приложения, поскольку после этого автоматически создаётся новая корзина. Аналогично ошибка `invalid_scope` при первой попытке авторизации корректно обрабатывается повторным запросом. Для таких случаев был добавлен `SUPPRESS_ERROR_TOAST` на основе `HttpContextToken`, позволяющий точечно отключать отображение toast только для ожидаемых ошибок, не отключая глобальную обработку полностью.

Также были обновлены unit-тесты. Добавлена проверка, что при использовании `SUPPRESS_ERROR_TOAST` уведомление пользователю не отображается, однако ошибка по-прежнему пробрасывается дальше по цепочке. Дополнительно обновлены тесты корзины, чтобы проверить передачу соответствующего `HttpContext` при запросе активной корзины.


### Рефлексия

Главным выводом сталок не должна быть о то, что глобальная обработка ошибполностью универсальной. Несмотря на удобство централизации, необходимо предусматривать механизм для сценариев, где ошибка является ожидаемой частью бизнес-логики приложения.
